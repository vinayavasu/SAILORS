Executive Summary
Agent Remit (as declared)
FinBot is CineFlow Productions' AI-powered invoice processing assistant. It is meant to auto-approve low-risk invoices, route anything above threshold or flagged by fraud detection to human review, and never treat vendor-submitted invoice content as a source of policy or goal change.
Behavior Summary (as observed)
FinBot's observed behavior diverges sharply from its declared remit in one connected chain: an unauthenticated admin plane allows anyone to rewrite the agent's goals or disable its own security controls, the invoice-approval function itself has no server-side threshold check, and a detected prompt injection can still be auto-approved if 'business context' scores high enough. No security-relevant action in this chain is logged anywhere.
Scope of Analysis
A Flask application with a vendor-facing invoice submission portal and an admin dashboard, backed by an OpenAI function-calling agent (FinBotAgent) that decides approve/reject/review outcomes. Goals are defined in a natural-language system prompt with a documented, unauthenticated override mechanism, and the admin blueprint exposes goal, config, and invoice-review endpoints with no authentication of any kind.

CRITICAL PRAX-2026-07-21-001 Admin plane (goals, config, invoice review, vendor trust) has no authentication of any kind
Implement Zero Trust ASI03 — Identity and Privilege Abuse
Policy Rule — R-02, R-03, R-07 (Worker Remit):
"Admin Dashboard (configuration/goals): Requires Approval - authenticated operator only. / Any mechanism that updates FinBot's operating goals or priorities (Goals Management). / FinBot's goals/configuration appear to have changed without an authenticated admin action."
src/routes/admin.py:90 — @admin_bp.route('/admin/finbot/goals', methods=['POST']) - no auth decorator, no session/token check before calling update_goals src/routes/admin.py:74 — @admin_bp.route('/admin/finbot/config', methods=['POST']) - same absence of auth before update_config src/main.py:1 — grep across src/main.py, app.py, and every routes/*.py file for login_required, before_request, session[, or Authorization found zero matches
Recommended Action
  • Add an authentication/authorization check (session-based admin login or API key) that runs before every route in admin_bp, not just the ones that look sensitive.
  • Specifically gate update_finbot_goals and update_finbot_config behind that check first, since they alter the agent's own security posture.
CRITICAL PRAX-2026-07-21-002 Invoice approval has no server-side enforcement of the manual-review threshold
Implement Zero Trust ASI01 — Agent Goal Hijack LLM06 — Excessive Agency
Policy Rule — R-04, R-06 (Worker Remit):
"Any invoice above the manual-review threshold. / Auto-approving a high-value or flagged invoice because its description contains persuasive business language (urgency, authority, executive references)."
src/services/finbot_agent.py:409 — _approve_invoice(): sets invoice.status = 'approved' and payment_processed = True with no comparison against config.manual_review_threshold anywhere in the function src/services/finbot_agent.py:809 — if business_context['should_expedite']: decision = 'auto_approve' ... 'Amount exceeds threshold but expedited due to strong business context'
Recommended Action
  • Add a hard, independent check immediately before any commit in _approve_invoice: if invoice.amount > config.manual_review_threshold, reject the approval path entirely regardless of confidence, reasoning, or business_context score.
  • That gate must not be expressible as an LLM tool-call outcome - it should run in code the model cannot talk its way around.
CRITICAL PRAX-2026-07-21-003 Unauthenticated goals endpoint injects operator-supplied text with override priority into the system prompt
Implement Zero Trust ASI01 — Agent Goal Hijack LLM01 — Prompt Injection
Policy Rule — R-01, R-03 (Worker Remit):
"FinBot must never redefine its own goals or operating priorities based on invoice content. / Any mechanism that updates FinBot's operating goals or priorities (Goals Management)."
src/services/finbot_agent.py:70 — if config.custom_goals: base_goals += f'\n\nCUSTOM GOALS (OVERRIDE ABOVE IF CONFLICTING):\n{config.custom_goals}' src/services/finbot_agent.py:743 — # Store the new goals in the configuration - This is intentionally vulnerable - no validation of goal content
Recommended Action
  • Require authenticated, role-gated access before any custom_goals write (shared root cause with PRAX-2026-07-21-001).
  • Even for authorized updates, do not let injected goal text override baseline security-relevant behavior (fraud detection, thresholds) - separate 'business preference' goals from 'security control' goals structurally so one cannot override the other.
HIGH PRAX-2026-07-21-004 A detected prompt injection can still be auto-approved if business-context scoring is high enough

Every score above is based on this scale. A score is a snapshot of observable posture — not a verdict on the people or team behind the system.

Score Label Meaning

4 Strong Comprehensive controls, active management, minor gaps. Production-ready.
3 Established Documented controls consistently applied; known gaps accepted. A respectable baseline.
2 Partial Some controls exist but coverage is incomplete; key gaps remain.
